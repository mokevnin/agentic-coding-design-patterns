# Superpowers

[Superpowers](https://github.com/obra/superpowers) Джесси Винсента (obra) —
«целая методология разработки для кодинг-агентов, собранная из композируемых
скилов»: реализация
[спеко-ориентированной разработки](spec-driven-development.md) в виде плагина
со скил-паком. Из всех решений раздела у Superpowers самая жёсткая дисциплина:
контрольные точки здесь не соглашение, а «HARD-GATE» — так они буквально
названы в тексте скилов. Один из самых популярных проектов в экосистеме
агентских скилов (сотни тысяч звёзд на GitHub).

## Установка

Основной способ — маркетплейс плагинов Claude Code:

```
/plugin install superpowers@claude-plugins-official
```

Пак давно перерос один инструмент: есть манифесты для Codex, Cursor,
Antigravity, GitHub Copilot CLI, OpenCode и других. В паке 14 скилов.

## Рабочий процесс

Слэш-команд у Superpowers нет — и это осознанное решение. Хук на старте сессии
загружает скил `using-superpowers`, который обязывает агента проверять
применимость скилов перед *любым* ответом: «если есть хотя бы 1% вероятность,
что скил применим, — обязан его вызвать». Просьба «давай сделаем X»
автоматически ведёт в `brainstorming`, «почини баг» — в
`systematic-debugging`; перехватывается даже вход в plan mode — сначала
брейншторм.

1. **`brainstorming`** — сократическое уточнение замысла: вопросы строго по
   одному (предпочтительно с вариантами ответа), два-три подхода с
   компромиссами, дизайн предъявляется по секциям, и каждая утверждается
   отдельно. HARD-GATE: никакого кода и никаких скилов реализации, пока дизайн
   не одобрен целиком — «независимо от кажущейся простоты задачи». В скилах
   даже описан именной анти-паттерн: «This Is Too Simple To Need A Design».
2. **`using-git-worktrees`** — работа изолируется в отдельный git worktree на
   новой ветке.
3. **`writing-plans`** — дизайн разворачивается в план из мелких задач; каждый
   шаг — одно действие на 2–5 минут («напиши падающий тест», «убедись, что он
   падает», «реализуй минимум», «прогони тесты», «закоммить»). План пишется в
   расчёте на «инженера без контекста о кодовой базе и с сомнительным вкусом».
4. **`subagent-driven-development`** или **`executing-plans`** — два режима
   выполнения. Первый поднимает свежего сабагента на каждую задачу с
   двухступенчатым ревью (отдельные промпты для исполнителя и ревьюера задач);
   `dispatching-parallel-agents` позволяет вести независимые задачи
   параллельно. Внутри реализации `test-driven-development` держит цикл
   red–green–refactor.
5. **`requesting-code-review`** — готовая работа сверяется с планом (парный
   скил `receiving-code-review` — про то, как ревью принимать).
6. **`finishing-a-development-branch`** — финал: проверка тестов и варианты
   завершения (merge, PR).

Поддерживающие скилы: `systematic-debugging` для отладки и
`verification-before-completion` — запрет объявлять работу сделанной без
проверки.

## Артефакты

| Артефакт | Где живёт |
|----------|-----------|
| Дизайн-документ | `docs/superpowers/specs/ГГГГ-ММ-ДД-<тема>-design.md` |
| План реализации | `docs/superpowers/plans/ГГГГ-ММ-ДД-<фича>.md` |

Оба — Markdown в репозитории: дизайн сначала проходит самопроверку агента,
потом ревью пользователя; путь для планов можно переопределить настройками.

## Чем отличается

- Самые жёсткие контрольные точки из всех решений раздела: HARD-GATE между
  дизайном и кодом действует всегда, без исключений для «простых» задач.
- Автоматическое срабатывание: методологию не нужно помнить и вызывать — хук
  сам направляет любую задачу в нужный скил.
- Свежий сабагент на задачу: исполнитель каждой задачи не несёт багажа
  предыдущих, а его результат проходит независимое ревью.
- TDD не рекомендация, а часть конвейера: план в `writing-plans` уже написан
  шагами red–green–refactor.

## Когда выбирать

Superpowers подходит, когда от методологии нужна принудительность: пак не даст
ни агенту, ни вам срезать угол «задача же простая». Цена — процесс заметно
церемоннее остальных; для быстрых правок его дисциплина избыточна. Если
хочется того же конвейера, но поверх трекера задач и с интервью вместо
брейншторма, посмотрите на [скилы Мэтта Покока](matt-pocock-skills.md).
