# Скилы Мэтта Покока

[Пак скилов Мэтта Покока](https://github.com/mattpocock/skills) — реализация
[спеко-ориентированной разработки](spec-driven-development.md) в виде набора
скилов Claude Code. В отличие от Spec Kit и OpenSpec, которые складывают
артефакты в каталог репозитория, этот пак строит конвейер поверх **трекера
задач**: спецификация публикуется как иссью, задачи — как связанные тикеты, и
дальше работа идёт по обычному для команды процессу.

## Установка и настройка

Два способа: редактируемый — `npx skills@latest add mattpocock/skills`
(скилы копируются в репозиторий, их можно форкать и править) — и управляемый,
через маркетплейс плагинов Claude Code
(`/plugin install mattpocock-skills@mattpocock`). Затем один раз на репозиторий
запускается `/setup-matt-pocock-skills`: скил-настройщик определяет, где живёт
трекер (GitHub по умолчанию, поддерживается и локальный вариант на
Markdown-файлах), словарь triage-меток (`needs-triage`, `ready-for-agent`,
`ready-for-human`…) и раскладку доменных документов; конфигурация оседает в
`docs/agents/`.

## Рабочий процесс

Основной поток «идея → продакшен» собирается из скилов по фазам SDD:

1. **Интервью вместо чернового промпта.** `/grill-me` безжалостно
   интервьюирует *вас*: по одному вопросу за раз, с рекомендуемым ответом на
   каждый; факты скил ищет в кодовой базе сам, а решения оставляет человеку.
   Вариант `/grill-with-docs` попутно ведёт доменную документацию через
   `domain-modeling`: глоссарий в `CONTEXT.md` и ADR фиксируются в момент,
   когда решение кристаллизовалось.
2. **Спецификация.** `/to-spec` синтезирует проработанный разговор в
   самодостаточную спецификацию (по сути PRD) — без нового интервью — и
   публикует её в трекер с меткой `ready-for-agent`. В спецификации намеренно
   нет путей к файлам и фрагментов кода — они устаревают быстрее всего.
   Отдельный шаг — выбор «швов» тестирования: скил предъявляет их пользователю
   на согласование, предпочитая существующие швы новым.
3. **Задачи.** `/to-tickets` режет спецификацию на **трассирующие тикеты** —
   вертикальные срезы «сквозь все слои: схема, API, UI, тесты», каждый
   размером в одно свежее контекстное окно и каждый явно объявляет, какие
   тикеты его блокируют. На настоящем трекере связи становятся нативными
   blocking-links, в локальном варианте — текстом в файле тикета. Правило
   выполнения — «работай по фронтиру»: бери любой тикет, все блокеры которого
   закрыты. Исключение — широкие механические рефакторинги: они не режутся на
   срезы, а выстраиваются по схеме expand–contract.
4. **Реализация.** `/implement` ведёт работу по спецификации или тикетам:
   внутри — `/tdd` на заранее согласованных швах, регулярный тайпчек и
   прогон тестов, в конце — полный сьют.
5. **Ревью.** `/implement` завершается `/code-review`: две оси — соответствие
   стандартам репозитория и соответствие исходной спецификации — проверяются
   параллельными сабагентами, чтобы не загрязнять контекст друг друга.

Вокруг основного конвейера — скилы для масштаба и краёв процесса:

- `/wayfinder` — когда работа больше одной спецификации: замысел
  разворачивается в карту исследовательских тикетов на трекере, и агент
  разрешает их по одному, пока путь не станет ясен.
- `/triage` — входящие иссью проходят через канонические метки
  (`needs-triage` → `ready-for-agent` / `ready-for-human`) и доводятся до
  брифа, пригодного для агента.
- `/prototype` — одноразовый прототип, отвечающий на дизайн-вопрос, который
  не решается разговором.
- `/handoff` — сжатие разговора в документ передачи для следующей сессии, с
  отсылками к уже созданным артефактам вместо их дублирования.

## Артефакты

| Артефакт | Где живёт |
|----------|-----------|
| Спецификация (PRD) | Иссью в трекере с меткой `ready-for-agent` |
| Тикеты с blocking-связями | Трекер (или файлы в `.scratch/<фича>/issues/`, если трекер локальный) |
| `CONTEXT.md` | Корень репозитория: доменный глоссарий |
| ADR | `docs/adr/`: архитектурные решения |
| Handoff-документ | Временный каталог ОС — намеренно вне репозитория |

## Чем отличается

- Конвейер поверх трекера: спецификация и тикеты живут там же, где задачи
  людей, — процесс команды не раздваивается.
- Интервью как первая фаза: пак исходит из того, что дыры в замысле дешевле
  всего найти *до* спецификации, разговором.
- Трассирующие тикеты: нарезка не на «слои» (модель, API, UI), а на
  вертикальные срезы с явным графом блокировок — каждый тикет доводит фичу до
  проверяемого состояния.
- Доменная документация как побочный продукт: глоссарий и ADR пишутся по ходу
  интервью, а остальные скилы обязаны использовать этот словарь.

## Когда выбирать

Пак Мэтта Покока — вариант для тех, кто уже живёт в Claude Code и трекере
задач и хочет SDD без нового инструмента в стеке: конвейер собирается из
слэш-команд, а артефакты попадают в привычные иссью. Если нужна жёсткая рамка
с фиксированными файлами-артефактами в репозитории, ближе
[Spec Kit](spec-kit.md) или [OpenSpec](openspec.md); философски же ближе всего
[Superpowers](superpowers.md) — другой скил-пак с более строгими контрольными
точками.
