# GitHub Spec Kit

[Spec Kit](https://github.com/github/spec-kit) — открытый тулкит GitHub и
самый прямой перенос
[спеко-ориентированной разработки](spec-driven-development.md) в инструмент:
слэш-команда на каждую фазу конвейера, артефакт на каждую команду. Именно
[анонс Spec Kit](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)
сделал аббревиатуру SDD общеупотребительной.

Spec Kit не привязан к конкретному агенту: одни и те же команды работают в
Claude Code, GitHub Copilot, Cursor, Gemini CLI, Codex CLI и ещё десятках
интеграций (полный список — `specify integration list`).

## Установка

Тулкит ставится CLI-утилитой `specify` (Python):

```sh
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git
specify init my-project --integration claude
```

`specify init` кладёт в проект каталог `.specify/` — шаблоны артефактов,
память проекта (`memory/constitution.md`) и расширения — и регистрирует
слэш-команды в выбранном агенте.

## Рабочий процесс

Фазы идут строго по порядку, каждая команда оставляет артефакт, который
разработчик ревьюит до перехода к следующей:

1. `/speckit.constitution` — соглашения проекта: принципы, стек, стандарты
   качества. Пишется один раз, действует на все фичи.
2. `/speckit.specify` — спецификация фичи: пользовательские истории и
   требования. Намеренно без технических решений: команда отказывается
   обсуждать стек — это дело плана.
3. `/speckit.clarify` (по желанию) — агент находит недоопределённые места
   спецификации, задаёт вопросы и вписывает ответы в неё же.
4. `/speckit.plan` — технический план: архитектура, контракты, модель данных.
5. `/speckit.tasks` — нарезка плана на задачи с зависимостями.
6. `/speckit.analyze` (по желанию) — сверка согласованности: не противоречат
   ли друг другу спецификация, план и задачи.
7. `/speckit.implement` — выполнение задач по списку.

Из более новых команд: `/speckit.checklist` генерирует чек-листы качества,
`/speckit.taskstoissues` превращает задачи в GitHub-иссью, а
`/speckit.converge` оценивает существующую кодовую базу против артефактов —
мостик для браунфилда.

## Артефакты

| Путь | Что лежит |
|------|-----------|
| `.specify/memory/constitution.md` | Соглашения проекта |
| `.specify/templates/` | Шаблоны артефактов (можно переопределять) |
| `specs/<номер>-<фича>/spec.md` | Спецификация фичи |
| `specs/<номер>-<фича>/plan.md` | Технический план |
| `specs/<номер>-<фича>/tasks.md` | Список задач |

Все артефакты — обычный Markdown в репозитории: их ревьюят, правят и коммитят
как код.

## Чем отличается

Точка зрения Spec Kit — «спецификации становятся исполняемыми»: намерение, а
не код, является источником истины. Отличительные черты:

- Жёсткое разделение «что» и «как»: спецификация не знает о стеке, план не
  переопределяет требования.
- Линейный конвейер с контрольной точкой после каждой команды — ближе всего к
  каноническому описанию паттерна.
- Экосистема расширений: `specify extension add`, `specify preset add`,
  `specify bundle install` — команды и шаблоны можно наращивать.
- Агент-агностичность: один процесс на всю команду, даже если люди работают в
  разных инструментах.

## Когда выбирать

Spec Kit хорош, когда нужен канонический SDD-конвейер «из коробки» и команда
работает в разных агентах. Слабое место — браунфилд: процесс изначально
рассчитан на «фичу с нуля», и хотя `/speckit.converge` сокращает разрыв,
работа от изменений в существующей системе естественнее ложится на
[OpenSpec](openspec.md).
