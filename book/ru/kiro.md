# Kiro

[Kiro](https://kiro.dev) — ответ AWS на
[спеко-ориентированную разработку](spec-driven-development.md): не тулкит
поверх чужого агента, а продукт, в который SDD встроена как режим. Kiro
существует в двух формах: IDE (форк Code OSS, запущен в середине 2025) и Kiro
CLI — переименованный Amazon Q Developer CLI. Агент здесь свой, на моделях
семейства Claude; чужие ассистенты Kiro не поддерживает.

## Рабочий процесс

Рядом с обычными вайб-сессиями в Kiro есть **спек-сессии**: интерфейс сам
ведёт через три фазы, и каждая требует явного одобрения разработчика — без
слэш-команд, кнопками IDE:

1. **Requirements** → `requirements.md` — пользовательские истории с
   критериями приёмки в EARS-нотации: «WHEN *условие* THEN the system SHALL
   *действие*». Такие формулировки можно проверить механически. Отдельная
   кнопка «Analyze Requirements» запускает углублённый анализ требований.
2. **Design** → `design.md` — архитектура, интерфейсы, диаграммы
   последовательности; Kiro генерирует дизайн, читая существующий код.
3. **Tasks** → `tasks.md` — план реализации с привязкой задач к требованиям.
   Задачи отмечаются по одной, а «Run all Tasks» строит граф зависимостей и
   выполняет независимые задачи параллельно — «волнами».

Входов в спек-сессию несколько: Requirements-First (классика),
Design-First (когда архитектура — главный вопрос) и Quick Spec (укороченный
вариант). Для багов есть отдельный тип спеки — bugfix-спека с `bugfix.md` и
формулировками вида «WHEN *условие* THEN the system SHALL CONTINUE TO
*существующее поведение*» — регрессия описывается как требование.

## Артефакты

| Путь | Что лежит |
|------|-----------|
| `.kiro/specs/<фича>/requirements.md` | Требования в EARS-нотации |
| `.kiro/specs/<фича>/design.md` | Технический дизайн |
| `.kiro/specs/<фича>/tasks.md` | План реализации |
| `.kiro/steering/` | Steering-файлы проекта |
| `~/.kiro/steering/` | Глобальные steering-файлы |
| `.kiro/hooks/` | Хуки-автоматизации (JSON) |

Роль соглашений проекта играют **steering-файлы** — правила, подмешиваемые в
каждую сессию. По умолчанию их три: `product.md` (что за продукт), `tech.md`
(стек и ограничения), `structure.md` (устройство репозитория). Режим
подключения задаётся front matter'ом: always (по умолчанию), fileMatch (по
маске файлов), manual (по упоминанию `#имя-файла`), auto.

Дополняют картину **хуки-агенты**: JSON-файлы с триггерами
(`PostFileSave`, `PreToolUse`, `SessionStart`, `Stop` и другие), которые
запускают автоматизации — например, обновить тесты после сохранения файла.

## Чем отличается

- SDD как режим продукта, а не дисциплина разработчика: интерфейс физически
  не даёт перескочить из требований в код — следующая фаза открывается только
  после одобрения предыдущей.
- EARS-нотация в требованиях: критерии приёмки формулируются проверяемо, а не
  прозой.
- Волновое выполнение задач: граф зависимостей строится автоматически,
  независимые задачи идут параллельно.
- Steering и хуки дают постоянный контекст и автоматизацию без внешних
  инструментов.

## Когда выбирать

Kiro подходит, когда хочется, чтобы дисциплину конвейера обеспечивал сам
инструмент, и нет привязки к другому агенту — цена в том, что Kiro закрытая
экосистема AWS. Если агент уже выбран (Claude Code, Cursor, Copilot),
агент-агностичные [Spec Kit](spec-kit.md) и [OpenSpec](openspec.md) встроятся
в существующий процесс без смены инструментов.
