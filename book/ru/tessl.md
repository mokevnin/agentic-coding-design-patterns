# Tessl

[Tessl](https://tessl.io) — самая радикальная трактовка
[спеко-ориентированной разработки](spec-driven-development.md): спецификация —
не спутник кода, а его *источник*. Код — производный артефакт, как бинарник
при компиляции; редактировать его в обход спецификации — всё равно что патчить
выхлоп компилятора. Компанию основал Гай Подъярный (основатель Snyk), под эту
ставку привлечено около 125 млн долларов.

Tessl агент-агностичен: заявлена работа со всеми популярными кодинг-агентами.

## От фреймворка к платформе

Tessl прошёл заметную эволюцию, и это стоит учитывать при чтении материалов о
нём:

- **Фреймворк и Spec Registry (2025).** В исходном фреймворке файлы кода
  помечались аннотациями: `@generate` — код генерируется из спецификации и
  руками не редактируется, `@describe` — спецификация документирует
  существующий код. Возможности (capabilities) спецификации привязывались к
  тестам, и тесты проверяли, что регенерированный код ей соответствует.
  Spec Registry — реестр с более чем 10 000 готовых спецификаций популярных
  открытых библиотек, чтобы агенты не галлюцинировали их API.
- **Платформа «context as code» (2026).** Сегодня Tessl позиционируется шире —
  как платформа контекста для агентов: реестр-пакетный менеджер, управление,
  евалы, наблюдаемость, собственный Tessl Agent. SDD стала одним из
  устанавливаемых плагинов платформы.

## Рабочий процесс

SDD-плагин ставится через CLI:

```sh
tessl init
tessl install tessl-labs/spec-driven-development
```

Дальше процесс выглядит так:

1. **Сбор требований.** Агент задаёт уточняющие вопросы, пока намерение не
   станет однозначным.
2. **Спецификации.** Агент пишет Markdown-спецификации в каталог `specs/`;
   возможности связываются с тестами пометками `[@test]` — у каждого
   утверждения спецификации есть проверка.
3. **Одобрение и реализация.** После ревью спецификаций агент реализует код
   против них — и вносит обратно в спецификации всё, что выяснилось в ходе
   разработки: документ и код не расходятся.

Вокруг этого — платформенные команды: `tessl search / install / list /
update` (работа с реестром), `tessl review run` (ревью, включая
security-проверки на движке Snyk), `tessl eval run` (евалы), `tessl skill new
/ publish` и `tessl plugin new / publish` (свои скилы и плагины).

## Чем отличается

- Инверсия отношения «спецификация — код»: не спецификация описывает код, а
  код реализует спецификацию; при конфликте правится код.
- Тесты привязаны к утверждениям спецификации: «спецификация выполняется» —
  это буквально зелёный прогон её проверок.
- Реестр как пакетный менеджер: спецификации, скилы и плагины
  распространяются и версионируются как пакеты.
- Максимализм подхода: остальные фреймворки делают спецификацию *первой*,
  Tessl — *главной*.

## Когда выбирать

Tessl — выбор для тех, кто готов принять ставку «код — производный артефакт»
целиком: она даёт самую сильную гарантию синхронности документа и кода, но
требует наибольшей перестройки привычек. Для более консервативного входа в
SDD лучше начать со [Spec Kit](spec-kit.md) или [OpenSpec](openspec.md) —
там спецификация дисциплинирует процесс, но код остаётся обычным кодом.
